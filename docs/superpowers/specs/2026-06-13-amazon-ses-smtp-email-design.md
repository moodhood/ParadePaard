# Amazon SES SMTP Email Migration

## Goal

Replace every current MailerSend-based outbound email path with Amazon Simple Email Service (SES) over SMTP while preserving existing application behavior and email content.

## Scope

The current outbound email implementations are:

- Auth service password-reset emails.
- Auth service employee onboarding emails.
- Auth service employee account-setup emails.
- Contract service contract-ready emails.

The user service and frontend only trigger these flows and do not send email directly. They require no provider-specific changes.

## Architecture

Both email-owning Spring Boot services will use Spring's `JavaMailSender` through `spring-boot-starter-mail`.

- SMTP host: `email-smtp.eu-north-1.amazonaws.com`
- SMTP port: `587`
- Transport security: STARTTLS enabled and required
- Authentication: enabled
- Sender: `noreply@lambdamanager.com`

The `lambdamanager.com` domain is verified in SES region `eu-north-1`, so the sender address is covered by that identity.

Each service keeps its existing application-facing interface:

- `EmailSender` in the auth service.
- `ContractEmailSender` in the contract service.

Only the provider adapters behind those interfaces change. This keeps password-reset, onboarding, account-setup, and contract workflows independent of SES-specific details.

## Configuration And Credentials

Runtime configuration will use:

- `SES_SMTP_HOST`, defaulting to `email-smtp.eu-north-1.amazonaws.com`.
- `SES_SMTP_PORT`, defaulting to `587`.
- `SES_SMTP_USERNAME`, with no committed default.
- `SES_SMTP_PASSWORD`, with no committed default.
- `SES_FROM_EMAIL`, defaulting to `noreply@lambdamanager.com`.

`SES_SMTP_USERNAME` is the SES SMTP username. `SES_SMTP_PASSWORD` must be the region-specific SES SMTP password generated by the SES console or derived using AWS's documented conversion process. The AWS secret access key must not be used directly as the SMTP password.

No credentials will be committed to source control. The ignored local `.env` may hold development credentials, while `.env.example` will contain names and blank placeholders only. Docker Compose will pass the variables into both email-owning services.

The AWS access key and secret disclosed in the conversation must be deactivated and replaced. The replacement should have only the permissions needed to send email, and its derived SMTP credentials should be used by the application.

## Email Delivery

The auth service adapter will create MIME messages containing the existing plain-text and HTML bodies and subjects. The contract service adapter will do the same for contract-ready messages.

Each message will set:

- From: configured SES sender.
- To: the workflow-provided recipient.
- Subject: the existing subject.
- Plain-text body: existing content.
- HTML body: existing content.

The current HTML escaping behavior for dynamic values remains in place.

## Error Handling

Missing SMTP credentials or an SES delivery failure must fail the send operation clearly.

- Auth email failures continue to propagate as runtime failures through the existing workflow behavior.
- Contract email failures continue to surface as `ContractEmailDeliveryException`.
- Logs identify the email purpose and sender/recipient but never include credentials or message secrets.
- Interrupted operations preserve the thread interrupt flag where applicable.

SES sandbox restrictions and identity errors are deployment concerns returned as delivery failures. Production sending requires the SES account in `eu-north-1` to have production access unless every recipient is verified.

## Removal

The migration removes:

- `MailerSendEmailSender`.
- `MailerSendContractEmailSender`.
- `MAILERSEND_API_TOKEN`.
- `MAILERSEND_API_URL`.
- Provider-specific HTTP and JSON request construction.
- MailerSend-specific tests and log/error wording.

Existing neutral interfaces and workflow services remain unchanged.

## Testing

Implementation follows test-driven development.

- Auth adapter tests verify sender, recipient, subject, plain-text body, HTML body, and provider failure propagation using a fake or mocked `JavaMailSender`.
- Contract adapter tests verify the existing review-and-sign copy and translation of mail failures to `ContractEmailDeliveryException`.
- Configuration tests or context tests verify that both services can construct their SES mail adapters with placeholder test credentials.
- Existing onboarding, password-reset, contract notification, and contract-send workflow tests remain green.
- Full Maven test suites run for auth service and contract service.

No test sends a real email or uses live SES credentials.

## Acceptance Criteria

- No tracked production code or configuration refers to MailerSend.
- Every current outbound email path uses SES SMTP.
- Both services use the same SES endpoint, credential variables, and sender variable.
- Credentials remain absent from tracked files and test output.
- Auth-service and contract-service tests pass.
